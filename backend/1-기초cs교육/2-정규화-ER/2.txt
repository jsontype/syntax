[Interface 설계시, ER 모델링 하기 체크리스트]

1. ER 모델링 절차
  - 1. 범위 정의 및 용어사전 확정 (업무 규칙, 데이터 개념 파악)
  - 2. 엔티티 정의 (업무상 독립적 존재, 생명주기와 식별자 확인)
  - 3. 속성 정의 (단순/복합/다중값/유도/식별자, 도메인 정의 포함)
  - 4. 키 설계 (후보키, 기본키, 대체키, 복합키/대리키 판단)
  - 5. 관계 정의 (이항/삼항, 카디널리티 1:1·1:N·N:M, 참여 제약 전체/부분)
  - 6. 관계 속성 검토 (관계 자체의 속성이 있는지 확인, 필요 시 연결 엔티티 승격)
  - 7. 약한 엔티티·연결 엔티티 정의 (식별/비식별 관계 여부 결정)
  - 8. 일반화/특수화(ISA) 설계 (슈퍼타입·서브타입, Disjoint/Overlap, Covering/Partial 명시)
  - 9. 무결성 제약 명시 (키, 도메인, 참조 무결성, 유일성, 체크 제약 → ERD에 설명)
  - 10. 개념 ERD 작성 (엔티티·속성·관계·제약 시각화)
  - 11. 논리 ERD 변환 (속성 정규화, PK/FK 지정, 다중값 속성 테이블 분리, 1:1·1:N·N:M 매핑 규칙 적용)
  - 12. 물리 스키마 설계 (테이블 생성 규칙, 컬럼 자료형, 인덱스 설계, 감사 속성 정의, 명명 규칙 적용)
  - 13. 성능 고려 (필요 시 역정규화, 집계 캐시, 중복 FK, 색인 최적화 등)
  - 14. 운영 규칙 정의 (FK ON DELETE/UPDATE 동작, 트랜잭션 경계, 데이터 사전·ERD 동기화)
  - 15. 검증 체크리스트 점검 (모든 엔티티 PK 존재 여부, 관계 카디널리티·참여 제약 명시 여부, 다중값/복합 속성 해소, 도메인 정의, 비즈니스 규칙과의 일치 여부)
  - 16. 최종 산출물 정리 (용어사전, 개념·논리·물리 ERD, 제약 목록, 명명 규칙, 변경 이력 정책)
  - 17. 이해관계자 리뷰 및 승인 (업무 담당자, DBA, 개발팀 등과 협의하여 최종 승인 받기)
